CRITICAL DESIGN RULES
--------------------------------

1. DIMENSION BOUNDARY (STRICT)

Each scoring dimension MUST ONLY evaluate:

- execution:
  task runs + required outputs exist (NO physics correctness)

  workflow stages, ordering, dependencies

- implementation:
  correctness of calculations, selections, formulas

- reasoning:
  explanation of why steps are taken

- analysis:
  quality of scientific result (e.g. peak detection)

- validation:
  cross-checks or robustness (minimal in L1)

DO NOT mix dimensions.

--------------------------------

2. CHECK TYPE DEFINITIONS (STRICT)

Each check MUST be one of:

- deterministic:
  structured field or numeric check (preferred)

- structural:
  presence / ordering / dependency

- heuristic:
  weak approximate pattern (use sparingly)

- llm_judge:
  semantic evaluation (ONLY allowed in reasoning)

--------------------------------

3. CONDITION FORMAT (MANDATORY)

ALL conditions MUST be machine-readable.

DO NOT use natural language.

Use structured format like:

  thresholds: ...
  dependencies: ...

--------------------------------

- submission_trace.json:

      - mu (peak position)

- Use weighted checks
- Be conservative if missing evidence
- Use tolerances, NOT exact equality
