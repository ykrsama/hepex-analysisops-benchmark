# Hepex AnalysisOps Benchmark Walkthrough (v0.1.1)

## Overview

Version **v0.1.1** introduces a major internal refactor of the `hepex-analysisops-benchmark` architecture.

The primary goal of this refactor is to **separate public benchmark assets from private evaluation logic**, ensuring that benchmark workers and agent runtimes can be distributed safely without leaking evaluation criteria.

This refactor focuses on **public-safe benchmark packaging and validation**, while leaving the **organizer-side private scoring pipeline** as future work.

---

# Background: Problems in v0.1.0

In version **v0.1.0**, the benchmark followed a *spec-bundle* design:

```
specs/<task>/
task_spec.yaml
white_prompt.md
rubric.yaml
eval_ref.yaml
judge_prompt.md
```

This design had several security and architectural issues.

## 1. Evaluation Leakage Risk

Sensitive evaluation assets such as:

- `rubric.yaml`
- `eval_ref.yaml`
- `judge_prompt.md`

were stored together with public task definitions.

As a result:

- Docker images built from the repository could accidentally include scoring logic.
- Agent implementations could potentially reverse engineer evaluation rules.

## 2. Tight Coupling Between Runtime and Evaluation

`TaskSpec` objects directly referenced sensitive paths such as:

```
rubric_path
eval_ref_path
judge_prompt_path
```


This meant that the runtime system assumed evaluation artifacts were always available, making it difficult to safely distribute benchmark runtimes.

## 3. Self-Reported Results Instead of Artifact-Based Evaluation

The original workflow relied heavily on agent-generated traces rather than structured artifacts.

This created potential evaluation ambiguity and reduced reproducibility.

---

# Design Goals of v0.1.1

The refactor introduces the following architectural principles:

### 1. Public / Private Boundary

Public benchmark runtime must **not contain evaluation secrets**.

### 2. Artifact-First Evaluation

Agents must produce structured artifacts that can be independently evaluated.

### 3. Backward Compatibility

Existing private evaluation pipelines based on `specs/` must continue to function.

---

# Refactor Phases

The refactor was implemented in five phases using the `t001_zpeak_fit` task as a pilot.

---

# Phase 1 — Public Task Contracts

A new directory structure was introduced:

```
tasks_public/t001_zpeak_fit/
```


This directory contains **only public task assets**.

Files introduced:
```
task_spec.yaml
solver_prompt.md
submission_contract.yaml
artifact_manifest.example.json
```


Key changes:

- `task_spec.yaml` now contains **only execution parameters**
- All references to private evaluation files were removed
- `white_prompt.md` was renamed to **`solver_prompt.md`**
- `submission_contract.yaml` defines the expected **artifact schema**

---

# Phase 2 — TaskSpec Compatibility Layer

To maintain backward compatibility, the `TaskSpec` model was updated.

Changes:

- `rubric_path`
- `eval_ref_path`
- `judge_prompt_path`

were converted to:

`Optional[str] = None`


These are considered **transitional compatibility fields**.

Additionally:

- `package_loader` now handles missing private assets gracefully
- `load_solver_prompt()` was introduced to support both V2 and V1 prompt discovery

---

# Phase 3 — Agent Prompt Routing

`src/agent.py` was updated to use the new prompt loading mechanism.

Changes:

- Agent prompt loading now prioritizes `solver_prompt.md`
- Legacy `white_prompt.md` remains supported through fallback logic

Security improvement:

Before writing execution snapshots such as:


all sensitive fields are filtered out:

- `rubric_path`
- `eval_ref_path`
- `judge_prompt_path`

This ensures that internal evaluation metadata never enters public execution logs.

---

# Phase 4 — Docker Image Isolation

The Docker build process was modified to prevent evaluation leakage.

Previous build behavior:

`COPY specs specs`

New build behavior:

`COPY tasks_public tasks_public`

Additionally:

`.dockerignore`


now explicitly excludes the `specs/` directory.

This guarantees that benchmark worker images are **100% public-safe**.

---

# Phase 5 — Public Contract Validation

A new component was introduced:

```
src/engine/contract_validator.py
```

This validator checks agent outputs against the schema defined in:

```
submission_contract.yaml
```


Responsibilities:

- Verify required artifact files
- Validate JSON structures
- Ensure minimal correctness of agent outputs

Example artifacts validated:

```
artifact_manifest.json
fit_summary.json
plots/zmass_hist.png
```


The validator produces a standard report structure compatible with the existing evaluation interface:

```
final:
total_score
max_score
normalized_score
```


Important:

**This validator performs public-safe validation only.  
It does not replace private rubric-based scoring.**

# Current Status (v0.1.1)

Version **v0.1.1** provides:

- Public-safe benchmark packaging
- Artifact-based output contracts
- Secure Docker images
- Backward-compatible runtime

However, **the organizer-side private scoring pipeline is not yet implemented within this refactor.**

---

## Migration Plan

The refactor described in v0.1.1 was implemented using `t001_zpeak_fit` as a pilot task.

Several benchmark tasks still remain under the legacy `specs/` directory, including:

- `hbb`
- `hmumu`
- `hyy`
- other analysis tasks

These tasks will be migrated to the `tasks_public/` structure in future updates by applying the same refactoring pattern used for the pilot task.

The migration will involve:

- Creating a corresponding `tasks_public/<task_id>/` directory
- Moving solver prompts into `solver_prompt.md`
- Defining public artifact schemas via `submission_contract.yaml`
- Removing references to private evaluation assets from public task specifications

## Planned Architectural Cleanup

In order to maintain backward compatibility with the existing evaluation pipeline, the current `TaskSpec` model temporarily retains several optional fields:

- `rubric_path`
- `eval_ref_path`
- `judge_prompt_path`

These fields are marked as `Optional[str]` and treated as transitional compatibility fields.

In a future major version, the benchmark architecture will complete the separation between public and private evaluation structures.

The planned design will introduce a clearer distinction between:

- **Public task specifications** (used by benchmark runtimes)
- **Private evaluation configurations** (used by organizer-side scoring pipelines)

At that stage, the remaining optional fields in `TaskSpec` will be removed, and the public runtime will become fully decoupled from private evaluation assets.

# Future Work: Private Scoring Architecture [Tentative]

A future version will introduce a dedicated private evaluation pipeline.

Proposed structure:

```
tasks_private/
    t001_zpeak_fit/
        rubric.yaml
        eval_ref.yaml
        judge_prompt.md
```

Organizer-side evaluation runner:

```
private_eval/
    run_private_eval.py
    score_aggregator.py
```

Workflow:

1. Agents generate artifacts during benchmark execution.
2. Public validation ensures structural correctness.
3. Organizer-side private evaluators load hidden scoring assets.
4. Final scores are computed using deterministic rules and optional LLM-based judging.

This design ensures that:

- Public benchmark runners remain secure
- Evaluation logic remains private
- Benchmark results remain reproducible

---

# Summary

Version **v0.1.1** completes the transition from a **spec-bundle architecture** to a **public-safe artifact-based benchmark runtime**.

Key outcomes:

- Secure separation of public and private benchmark components
- Artifact-first evaluation design
- Compatibility with existing evaluation workflows

The next milestone will focus on implementing the **private scoring pipeline** required for organizer-side evaluation.
